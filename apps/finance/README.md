# Finance Module

Comprehensive financial analytics for trading systems following the canonical **HaruQuant TradeRecord schema**.

> This README has been updated to be **fully consistent with the current source code** (`metrics.py`, `returns.py`, `drawdowns.py`, `ratios.py`, `risks.py`, `benchmark.py`, `distributions.py`, `efficiency.py`).
> Updated: 2026-01-01

## Design Principles

- **Pure functions**: No mutation of inputs; no global state.
- **Composable**: Trade-based → build equity → derive returns → compute ratios/risk.
- **Dashboard-ready**: Outputs are easy to render as KPIs, tables, charts.
- **Reproducible by default**, with a note:
  - Most functions are deterministic.
  - **Exceptions**: `risks.risk_of_ruin()` (Monte Carlo) and `risks.value_at_risk()` when using `"parametric"` or `"cornish_fisher"` can introduce randomness unless you control NumPy’s RNG seed.

## Quick Start

```python
import pandas as pd

from apps.finance import metrics, returns, drawdowns, ratios, risks

trades_df: pd.DataFrame = ...

# 1) Trade-based metrics
print("Win rate:", metrics.win_rate(trades_df))
print("Profit factor:", ratios.profit_factor(trades_df))
print("Expectancy:", ratios.expectancy(trades_df))
print("SQN:", metrics.sqn(trades_df))

# 2) Build equity from trades, then compute returns & drawdowns
equity = returns.equity_curve(trades_df, initial_balance=10_000)
ret_series = returns.returns_series(equity)

print("CAGR (%):", returns.cagr(equity))
print("Max DD (%):", drawdowns.max_strategy_drawdown_percent(equity))

# 3) Ratios & risk (from returns series)
print("Sharpe:", ratios.sharpe_ratio(ret_series))
print("Sortino:", ratios.sortino_ratio(ret_series))
print("VaR 95%:", risks.value_at_risk(ret_series, confidence=0.95, method="historical"))
```

## Data Requirements

### A) Trades DataFrame (`pd.DataFrame`)

**Minimal columns (work for most trade-based KPIs):**
- `profit_loss` (float) — net P&L per trade
- `close_time` (datetime) — trade close timestamp

**Highly recommended columns (unlock most analytics and proper UI splits):**
- `open_time` (datetime) — needed for time-in-market and exposure functions
- `type` (str) — `"buy"` or `"sell"` (**required** for All/Long/Short reporting)
- `size` (float) — position size (lots/contracts)
- `time_in_trade` (float) — duration in hours (optional if `open_time/close_time` exist)
- `commission`, `slippage`, `swap` (float) — cost breakdown used by cost metrics
- `r_multiple` (float) — R-multiple per trade
- `mfe_usd`, `mae_usd` (float) — excursion in USD (efficiency & MAE/MFE ratios)
- `mfe_pips`, `mae_pips` (float) — excursion in pips (if you track pips)
- `initial_risk_usd` (float) — initial risk per trade (unit-risk efficiency)
- `exit_reason` (str) — optional; useful for filters/segment reporting

### B) Equity Curve (`pd.Series`)

Several functions take an **equity curve**:
- Index: `DatetimeIndex`
- Values: equity/balance level (float)

### C) Returns Series (`pd.Series`)

Many ratios and risk metrics take a **percentage returns series**:
- `ret_series = returns.returns_series(equity_curve)`

### D) Benchmark Inputs

`benchmark.py` expects aligned strategy/benchmark series (or a DataFrame) with:
- `strategy`
- `benchmark`

## Notes on Known Source-Code Quirks

### Duplicate function definitions in `ratios.py`

The current `ratios.py` defines these functions **twice**:
- `upside_potential_ratio()`
- `calmar_ratio()`

In Python, the **last definition wins** (earlier one is overwritten at import time).
This README lists them as part of the public API, but for clarity you should remove the duplicates in source.

### Determinism and Monte Carlo

If you need strict reproducibility across runs:
- Prefer `risks.value_at_risk(..., method="historical")`.
- If you use `risk_of_ruin()` or non-historical VaR methods, set a NumPy seed at the application layer:

```python
import numpy as np
np.random.seed(123)
```

## Best Practices

1. **Use trades DataFrames for trade-based metrics** (win rate, expectancy, streaks, SQN).
2. **Use equity curves for drawdowns and period returns** (drawdown series, daily/monthly returns).
3. **Use returns series for ratios and tail risk** (Sharpe, Sortino, VaR/CVaR).
4. **Always compute All/Long/Short** using the same filtering rules to keep the UI consistent.
5. **Validate required columns** before calling functions that depend on them.
